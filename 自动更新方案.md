# 交易纪律工作台自动更新方案

## 现在采用的方案

桌面版使用 Electron 的自动更新机制：

1. 软件启动后 6 秒自动检查新版本。
2. 如果发现新版本，弹窗提示用户是否下载。
3. 用户确认后后台下载更新包。
4. 下载完成后提示「重启并安装」。
5. 用户确认后自动重启并安装新版。

## 发布渠道

使用 GitHub Releases 作为更新源。

- 普通提交到 `main`：只生成测试安装包，不推送给已安装用户。
- 推送版本标签，例如 `v1.0.1`：自动生成正式 Release，并让已安装用户收到更新提示。

## 每次发新版的流程

1. 修改代码。
2. 修改 `package.json` 里的版本号，例如：

   ```json
   "version": "1.0.2"
   ```

3. 同步锁定文件：

   ```bash
   npm install --package-lock-only
   ```

4. 提交代码：

   ```bash
   git add .
   git commit -m "Release v1.0.2"
   ```

5. 打版本标签：

   ```bash
   git tag v1.0.2
   ```

6. 推送到 GitHub：

   ```bash
   git push origin main
   git push origin v1.0.2
   ```

7. GitHub Actions 会自动创建 Release，并上传 Mac / Windows 安装包和自动更新索引文件。

## 数据不会因为升级丢失

当前桌面版固定使用本地端口 `5199`，避免每次启动端口变化导致浏览器本地数据识别不一致。

用户升级前仍建议先在「数据管理」里点一次「导出备份」。

## 私有仓库注意事项

如果 GitHub 仓库是私有仓库，普通用户的软件无法直接读取 Release 更新信息，除非在客户端内置访问 token。

不建议把 token 内置进客户端，因为安装包会发给朋友，token 有泄露风险。

更推荐的做法：

1. 测试阶段：手动下载安装包，或者把仓库改为 public。
2. 正式阶段：使用公开的更新仓库、对象存储、服务器更新源，或者 Apple / Microsoft 正式分发渠道。

## Mac 签名和公证

现在的 Mac 包是测试包，没有 Apple Developer ID 签名和公证。

因此 Mac 用户可能看到：

- “无法验证开发者”
- “应用已损坏”

短期可手动清除隔离标记解决。正式自动更新建议申请 Apple Developer 账号，并配置：

- Developer ID Application 证书
- Apple notarization
- hardened runtime

这一步完成后，Mac 自动更新体验才会接近正式商业软件。
